# 统计口径与报表规则

这部分最容易“看起来差不多，实际不一样”。当前系统至少有三套统计视角：

- 交易明细 / 汇总 / 图表
- 利润报表
- 分润结算净额

它们共享同一批交易数据，但不完全共用同一套公式。后续改动时，必须先区分自己要改的是哪一层口径。

## 1. 统计能力分层

### 1.1 交易层统计

来源：

- `GET /api/transactions`
- `GET /api/transactions/summary`
- `GET /api/transactions/charts`

职责：

- 以“账本运营视角”看数据
- 更偏向明细分布、状态结构、待到账风险、来源构成

### 1.2 利润报表层统计

来源：

- `GET /api/reports/profit`
- `GET /api/reports/profit/details`

职责：

- 以“利润核算视角”看数据
- 强调主营收入、主营支出、流量、平台抽成、关闭交易等指标

### 1.3 分润结算层统计

来源：

- `apps/api/src/lib/settlement.ts`

职责：

- 以“能拿去分润的净额”看数据
- 强调累计净额、已分润额、留存、分配明细

> 分润结算不在本文详细展开，本文主要关注交易页与报表页；分润详见 `04-分润与结算规则.md`。

## 2. 统一过滤维度

### 2.1 利润报表的过滤维度

`apps/api/src/lib/profit-report.ts` 中支持：

- `start`
- `end`
- `billAccount`

并且基础查询只会覆盖以下分类：

- `main_business`
- `manual_add`
- `traffic_cost`
- `platform_commission`
- `closed`
- `business_refund_expense`

### 2.2 交易页的过滤维度

`apps/api/src/routes/transactions.ts` 中支持：

- `start`
- `end`
- `billAccount`
- `orderId`
- `category`
- `status`
- `direction`
- `sourceType`

这里的过滤更加偏运营检索，而不是利润核算。

## 3. 交易汇总卡片口径

接口：`GET /api/transactions/summary`

实现方式：

- 按 `direction + status` 做 `groupBy`
- 输出：
  - `totalCount`
  - `incomeCount`
  - `expenseCount`
  - `successCount`
  - `pendingCount`
  - `incomeAmount`
  - `expenseAmount`
  - `netAmount`

### 3.1 公式

- `pendingCount = totalCount - successCount`
- `netAmount = incomeAmount - expenseAmount`

### 3.2 重要事实

- 这层没有排除 `internalTransfer`
- 这层也没有主营 / 非主营的概念
- 它是“当前筛选结果下的原始账本统计”，不是利润统计

## 4. 图表页口径

接口：`GET /api/transactions/charts`

### 4.1 图表页输出维度

当前会返回以下聚合结果：

- `byCategory`
- `byBillAccount`
- `byDay`
- `byStatus`
- `byDirection`
- `byStatusDirection`
- `pendingAging`
- `pendingOverview`
- `pendingExpectedByDay`
- `pendingByBillAccount`
- `byAccountCategory`
- `bySourceType`
- `keyRatios`
- `byClosedRefundDay`
- `settlementOverview`

### 4.2 通用聚合方式

图表页直接拉全量符合筛选的 `QualifiedTransaction`，逐条循环聚合。

因此它天然具备：

- 账号维度
- 分类维度
- 日期维度
- 来源维度
- 状态 / 收支交叉维度

### 4.3 待到账分析口径

“待到账”只看以下记录：

- `category in (main_business, manual_add)`
- `direction = income`
- `status in`
  - `等待对方确认收货`
  - `等待发货`
  - `等待对方付款`
  - `等待确认收货`

派生规则：

- 自动确认周期固定按 10 天
- `expectedArrivalTime = transactionTime + 10 天`
- `remainingDays = ceil((expectedArrivalTime - now) / 1天)`

由此得到：

- 老化桶：
  - `0-1天`
  - `2-3天`
  - `4-7天`
  - `8-14天`
  - `15天以上`
- 即将到期：
  - 1 天内
  - 3 天内
  - 7 天内
- 已超期：
  - `remainingDays < 0`

### 4.4 图表页关键收益率口径

图表页会在本地聚合出这些核心数：

- `mainSettledIncome`
- `mainPendingIncome`
- `mainExpense`
- `trafficCost`
- `platformCommission`
- `mainClosedAmount`
- `mainClosedIncome`
- `mainClosedExpense`
- `businessRefundExpense`

随后计算：

```text
closedDirectionDelta = mainClosedIncome - mainClosedExpense

pureProfitSettled =
  mainSettledIncome
  - mainExpense
  - trafficCost
  - platformCommission
  - businessRefundExpense
  + closedDirectionDelta

pureProfitWithPending =
  pureProfitSettled + mainPendingIncome
```

以及比率：

- `pendingIncomeRate = mainPendingIncome / (mainSettledIncome + mainPendingIncome)`
- `trafficCostRate = trafficCost / (mainSettledIncome + mainPendingIncome)`
- `platformCommissionRate = platformCommission / (mainSettledIncome + mainPendingIncome)`
- `closedAmountRate = mainClosedAmount / (mainSettledIncome + mainPendingIncome)`
- `refundAmountRate = businessRefundExpense / (mainSettledIncome + mainPendingIncome)`
- `pureProfitSettledRate = pureProfitSettled / mainSettledIncome`
- `pureProfitWithPendingRate = pureProfitWithPending / (mainSettledIncome + mainPendingIncome)`

### 4.5 图表页关于关闭交易的事实

图表页 `keyRatios` 当前总是直接使用：

```text
mainClosedIncome - mainClosedExpense
```

它**没有读取** `PROFIT_INCLUDE_CLOSED_DIRECTION_IN_PROFIT` 环境变量。

## 5. 利润报表页口径

来源模块：`apps/api/src/lib/profit-report.ts`

### 5.1 主指标定义

主指标枚举：

- `main_settled_income`
- `main_pending_income`
- `main_expense`
- `traffic_cost`
- `platform_commission`
- `main_closed`
- `business_refund_expense`

### 5.2 待到账状态

利润报表把以下状态视为主营待到账：

- `等待对方确认收货`
- `等待发货`
- `等待对方付款`
- `等待确认收货`

### 5.3 报表汇总规则

报表层会显式跳过：

- `internalTransfer = true`

并汇总出：

- 主营已到账收入
- 主营待到账收入
- 主营支出
- 流量消耗
- 平台抽成
- 主营关闭总额
- 主营关闭收入额
- 主营关闭支出额
- 主营关闭不计收支额
- 主营退款支出

### 5.4 报表层纯收益公式

```text
closedNetContribution =
  (env 开启时) mainClosedIncome - mainClosedExpense
  (env 关闭时) 0
```

然后：

```text
pureProfitSettled =
  mainSettledIncome
  - mainExpense
  - trafficCost
  - platformCommission
  + closedNetContribution

pureProfitWithPending =
  mainSettledIncome
  + mainPendingIncome
  - mainExpense
  - trafficCost
  - platformCommission
  + closedNetContribution
```

### 5.5 一个非常重要的现实差异

虽然报表也统计 `businessRefundExpense`，但当前 `pureProfitSettled` / `pureProfitWithPending` **不会额外扣减**它。

源码里还保留了兼容性注释：

- “Keep existing report 口径兼容：pureProfit 不额外扣除 businessRefundExpense。”

这意味着：

- 报表页能看到退款支出指标
- 但纯收益指标本身不减它

## 6. 利润报表明细钻取

接口：`GET /api/reports/profit/details`

做法：

- 先用 `buildProfitBaseWhere(filters)` 锁住基础业务范围
- 再叠加 `metricWhere(metric)` 锁住具体指标

例如：

- `main_settled_income`
  - `category in (main_business, manual_add)`
  - `direction = income`
  - `status = 交易成功`
- `main_pending_income`
  - `category in (main_business, manual_add)`
  - `direction = income`
  - `status in 待到账状态集合`
- `main_closed`
  - `category = closed`

## 7. 当前三层口径差异总表

| 维度 | `/transactions/summary` | `/transactions/charts keyRatios` | `/reports/profit` | `settlement.ts` |
| --- | --- | --- | --- | --- |
| 视角 | 账本总览 | 运营图表/运营收益率 | 利润报表 | 分润净额 |
| 是否显式排除 `internalTransfer` | 否 | 否（但核心收益率不统计该分类） | 是 | 是 |
| 是否统计 `businessRefundExpense` 指标 | 否 | 是 | 是 | 是 |
| 纯收益/净额是否扣退款支出 | 不适用 | 是 | 否 | 是 |
| 关闭交易方向差额是否受环境变量控制 | 不适用 | 否 | 是 | 是 |
| 是否把待到账收入纳入主收益指标 | 不适用 | `pureProfitWithPending` 会纳入 | `pureProfitWithPending` 会纳入 | 仅分润口径按策略处理 |

这是当前系统最重要的“口径分叉点”。

`apps/api/test/profit-report-baseline.test.ts` 以两份参考账单做了当前报表基线校验，现行结果为：

- `mainSettledIncome = 40628.6`
- `mainPendingIncome = 44270.4`
- `mainExpense = 545.2`
- `trafficCost = 8426.42`
- `platformCommission = 393.23`
- `mainClosedAmount = 5687.6`
- `pureProfitSettled = 32580.05`
- `pureProfitWithPending = 76850.45`

如果未来改了分类、过滤、纯收益公式，这组数字大概率会变化，应该把它视为“必须重新确认”的回归锚点。
