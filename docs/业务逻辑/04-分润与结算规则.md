# 分润与结算规则

本文档只描述当前代码已经实现并通过测试的分润行为。

## 1. 分润者配置

来源：`apps/api/src/lib/profit-participants.ts`

- 至少保留一个分润者
- 名称不能为空
- `billAccount` 可为空
- 一个 `billAccount` 只能绑定一个分润者
- 分润比例总和必须等于 `1`
- 比例按 `6` 位小数存储

## 2. 结算策略

来源：`apps/api/src/lib/settlement.ts`

- `cumulative`
- `incremental`

当前前端主入口仍以累计分润为主，增量能力保留在后端。

## 3. 分润净额口径

### 3.1 累计分润净额

累计净额使用截至结算时点的全量业务数据，公式为：

```text
累计净额 =
  主营已到账收入
  - 主营支出
  - 流量消耗
  - 平台抽成
  - 主营退款支出
  + 关闭交易方向净额（可开关）
```

其中：

```text
关闭交易方向净额 = 关闭收入 - 关闭支出
```

环境变量：

- `PROFIT_INCLUDE_CLOSED_DIRECTION_IN_PROFIT`

默认：

- `true`

### 3.2 增量分润净额

增量分润会先筛出未被增量分润标记过的候选交易，再按和累计分润一致的结构计算净额。

## 4. 当前累计分润公式

这是当前代码最核心的行为。

定义：

- `C_t`：当前结算时点累计净额
- `S_{t-1}`：上一次累计已分润额
- `r_t`：本次留存比例

则当前代码按下面公式计算：

```text
当前累计目标已分润额 T_t = max(0, C_t) × (1 - r_t)
本次应分 P_t = max(0, T_t - S_{t-1})
本次留存 L_t = max(0, C_t - (S_{t-1} + P_t))
```

等价理解：

- 每次都重新看当前全量账本
- 先算“截至当前理论上总共应该已分出去多少”
- 再减去“历史上已经真实分出去多少”

这与旧版 `v0.10` 的累计目标补差逻辑保持一致。

## 5. 负向场景

当前实现中：

- 如果 `T_t <= S_{t-1}`
- 则本次 `paidAmount = 0`
- 不创建新的正向应分金额

也就是说：

- 当前不会产生“负分润付款”
- 负差额只会体现在“本次无可分金额”

## 6. 留存含义

当前实现下，留存不是一个必须下次单独释放的现金池。

它的含义是：

- 在本次结算时，当前累计净额里有一部分暂时不分
- 到下一次时，系统会基于新的全量账本重新计算累计应分目标

因此：

- 上一轮留存不会被作为独立状态量强制加到下一轮可分金额里

## 7. 个人层分配

当前代码在个人层做了“累计差量”处理。

### 7.1 本批应分金额

```text
participant.amount = 本次批次应分金额 × ratio
```

做两位小数与尾差校正。

### 7.2 绑定账号已持有金额

系统会先按绑定账号在当前结算时点的累计净额占比，算出该账号截至当前“目标已持有金额”。

然后减去该分润者历史累计 `accountHeldAmount`，得到本批新增的 `accountHeldAmount`：

```text
本批 accountHeldAmount =
  当前累计目标持有金额
  - 历史累计已记录 held
```

### 7.3 本批实际应转金额

```text
actualTransferAmount = amount - accountHeldAmount
```

解释：

- 正数：本次还应给该分润者
- 负数：该分润者当前账号已持有过多，本次应回吐 / 少拿

## 8. 有效批次

每个分润批次都会写入：

- `cumulativeNetAmount`
- `cumulativeSettledAmount`
- `carryForwardAmount`
- `isEffective`

系统会按 `(strategy, billAccount)` 维度维护当前有效批次，并把它作为下一次累计分润的已分基线。

## 9. 当前实现的稳定边界

当前代码和测试已经稳定支持：

- 累计目标补差结算
- 成本类退款冲减净额
- 历史已持有差量分配
- 账号绑定与未绑定分润者混合分配
