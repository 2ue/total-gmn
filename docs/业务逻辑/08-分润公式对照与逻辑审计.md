# 分润公式对照与逻辑审计

本文档只做分析，不改实现。

目标有三个：

- 判断当前代码实现是否存在会导致分润失真的逻辑缺口
- 判断“每次按全量累计应分润减上次累计已分润”的业务公式是否更科学
- 把“业务公式”和“当前代码公式”整理成可直接对照的一张表

---

## 1. 审计结论

### 1.1 总体结论

当前 `cumulative` 分润主路径：

- **确实是全量累计计算**
- **在“当前留存只作用于本期差额、上期留存本次全部释放”这一政策前提下，主公式整体是自洽的**

也就是说，当前实现不是“完全错误”，而是：

- **实现了一种特定的留存政策**
- 而不是一个更通用、更清晰的“累计目标分润”模型

### 1.2 是否存在会导致分润不对的逻辑缺口

结论：**存在，而且不止一处。**

当前最值得关注的缺口有四类：

1. **有效批次基线缺口**
   - 下一次累计分润依赖“当前有效批次”
   - 但系统没有把“按时间最新的结算批次必须是有效批次”做成硬约束
   - 这会直接影响下一次累计分润基线

2. **负向差额与留存的主路径缺口**
   - 主预览/创建路径在负向差额下仍直接使用：
     - `carryForwardAmount = periodNetAmount * carryRatio`
     - `paidAmount = distributableAmount`
   - 这会产生负留存、负实发一类业务上不稳定的结果

3. **个人分配不是按“累计应得 - 累计已得”核算**
   - 当前每个人的 `actualTransferAmount`
   - 不是从“个人历史累计应得 / 已拿”台账推出来的
   - 而是基于“本批金额 + 当前绑定账号净额占比”估算出来的

4. **实现中存在未接入主路径的结算函数**
   - 一部分更接近“负值保护”的 helper 存在
   - 但主预览/创建路径并没有统一走它
   - 这会导致代码内部存在两套结算语义

### 1.3 实际测试是否支持当前实现

结论：**部分支持，但不能证明当前实现在所有关键场景下都正确。**

现有测试能支持的事实：

- 当前系统确实按“上次留存本次全部释放，只对本期差额应用留存比例”运行
- 当前系统确实支持全量累计口径下的分润预览
- 成本类退款的净额处理已被测试覆盖

现有测试不能支持的事实：

- “负向差额 + 正留存比例”在主路径中是否合理
- 历史更正后，个人级实际转账是否等价于“累计应得 - 累计已得”
- 后补历史批次 / 回填旧日期批次后，下一次累计分润基线是否仍然正确
- 多个历史批次时间穿插时，有效批次是否仍然是正确基线

## 2. 当前代码实际实现的分润公式

下面只看 `cumulative` 主路径，也就是 `apps/api/src/lib/settlement.ts` 中真正用于预览与创建批次的路径。

### 2.1 变量定义

定义：

- `C_t`
  - 当前结算时点的累计净额
- `C_{t-1}`
  - 上一个有效批次记录的累计净额
- `S_{t-1}`
  - 上一个有效批次记录的累计已分润额
- `L_{t-1}`
  - 上一个有效批次记录的留存额
- `r_t`
  - 本次留存比例
- `Δ_t = C_t - C_{t-1}`
  - 本期差额

### 2.2 当前代码主公式

当前代码主路径等价于：

```text
Δ_t = C_t - C_{t-1}

本期新增实发部分 = Δ_t × (1 - r_t)
本期新增留存部分 = Δ_t × r_t

本次应结金额 P_t = L_{t-1} + Δ_t × (1 - r_t)
本次留存 L_t = Δ_t × r_t

新的累计已分润 S_t = S_{t-1} + P_t
```

源码对应位置：

- 累计净额 `C_t`：`apps/api/src/lib/settlement.ts:864`
- 上次累计净额 `C_{t-1}`：`apps/api/src/lib/settlement.ts:860`
- 上次累计已分润 `S_{t-1}`：`apps/api/src/lib/settlement.ts:872`
- 上次留存 `L_{t-1}`：`apps/api/src/lib/settlement.ts:873`
- 本期差额 `Δ_t`：`apps/api/src/lib/settlement.ts:874`
- 本期新增实发：`apps/api/src/lib/settlement.ts:875`
- 本期留存：`apps/api/src/lib/settlement.ts:876`
- 本次应结：`apps/api/src/lib/settlement.ts:877`
- 新累计已分润：`apps/api/src/lib/settlement.ts:879`

### 2.3 当前代码隐含的留存政策

当前代码不是把留存看成“累计余额池”的通用定义，而是隐含了下面这条政策：

- **上次留存本次全部释放**
- **本次只对本期差额应用新的留存比例**

这条政策已经被现有测试直接锁定：

- `apps/api/test/settlement-allocation.test.ts:90`

## 3. 你提出的业务公式

你提出的口径可以整理成：

### 3.1 业务定义

- 每次分润都以**当前时点的全量累计账本**为准
- 先算“当前总应分润”
- 再减去“上次累计已分润”
- 留存不应该在结算时被重复单独加减，而应该体现在“当前总应分润”定义里

### 3.2 抽象公式

定义：

- `T_t`
  - 当前时点的**累计目标已分润额**
- `S_{t-1}`
  - 上次累计已分润额

则：

```text
本次应结 P_t = T_t - S_{t-1}
```

如果进一步定义：

- `C_t`
  - 当前累计净额
- `R_t`
  - 当前应保留的累计准备金余额

则：

```text
T_t = C_t - R_t
P_t = (C_t - R_t) - S_{t-1}
```

## 4. 两套公式的一一对照表

| 维度 | 你的业务公式 | 当前代码公式 | 是否等价 | 说明 |
| --- | --- | --- | --- | --- |
| 计算口径 | 全量累计账本 | 全量累计账本 | 是 | 两者都不是单纯新增流水口径 |
| 当前累计净额 | `C_t` | `cumulativeNetAmount` | 是 | 当前代码确实先重算截至当前的累计净额 |
| 上次已分润基线 | `S_{t-1}` | `previous batch.cumulativeSettledAmount` | 是 | 当前代码确实以“累计已分润额”作为一部分基线 |
| 本次应结主表达式 | `P_t = T_t - S_{t-1}` | `P_t = L_{t-1} + Δ_t × (1-r_t)` | 条件等价 | 只有在 `T_t` 的定义与当前留存政策一致时才等价 |
| 当前总应分润 `T_t` | 明确业务定义 | 代码未显式命名 | 否 | 当前代码没有把 `T_t` 抽象成显式变量 |
| 留存定义 | `R_t` 应是“当前累计准备金余额” | `L_t = Δ_t × r_t` | 否 | 当前代码只保留“本期差额留存” |
| 上次留存处理 | 应内化在 `T_t` 中 | 本次全部释放 `L_{t-1}` | 条件等价 | 当前代码相当于假定上次留存到本次必须全释放 |
| 留存作用范围 | 取决于 `R_t` 定义 | 仅作用于本期差额 `Δ_t` | 否 | 当前代码是特定政策，不是通用模型 |
| 历史修正/退款冲回 | 通过 `C_t` 重算后进入补差 | 通过 `Δ_t` 进入补差 | 是 | 在全量累计模型下方向一致 |
| 个人应得金额 | 应从累计目标和历史已分配可审计推出 | 按本批 `paidAmount × ratio` | 部分不等价 | 当前代码是批次分配，不是累计个人台账差额 |
| 个人已拿/应扣 | 应基于历史累计分配台账 | 基于当前账号净额占比估算 | 否 | 这是当前代码最明显的个人层缺口 |

## 5. 哪一种更科学合适

### 5.1 结论

如果目标是：

- 公式稳定
- 审计容易
- 历史更正后仍能清楚解释
- 留存政策可以被明确表达

那么更科学的做法是：

```text
先定义当前累计目标已分润额 T_t
再用 P_t = T_t - S_{t-1}
```

也就是：

```text
T_t = C_t - R_t
P_t = (C_t - R_t) - S_{t-1}
```

### 5.2 为什么更科学

原因不是“它更短”，而是它把三个概念完全拆开了：

1. 当前累计净额 `C_t`
2. 当前累计准备金余额 `R_t`
3. 已经累计分出去的金额 `S_{t-1}`

这样做的好处是：

- 留存政策可以独立变化
- 历史更正不会把留存语义混进“本期差额”
- 可以直接定义系统的审计不变量：
  - `当前累计目标已分润额 = 当前累计净额 - 当前累计准备金余额`
  - `本次应结 = 当前累计目标已分润额 - 上次累计已分润`

### 5.3 当前代码公式适合什么前提

当前代码只有在下面这个政策前提下才是最合理的：

- 留存只作用于本期差额
- 上次留存到本次必须全部释放
- 不需要把留存视为一个跨期独立维护的累计准备金余额

如果这正是你要的业务规则，那么当前主公式本身可以继续使用。

如果不是，那么更推荐把公式升级成：

```text
P_t = T_t - S_{t-1}
```

并显式定义 `T_t`。

## 6. 当前实现的逻辑审计结果

### 6.1 通过审计的部分

以下部分当前可以认为是成立的：

- 系统确实按全量累计净额重算
- 历史更正、退款、手续费退回会通过新的累计净额进入补差
- 在“上次留存本次全释放”的政策下，当前主公式对正向差额是闭合的
- 当前测试已经覆盖：
  - 上次留存本次释放
  - 本期差额应用留存比例
  - 成本类退款冲减净额

### 6.2 发现的逻辑缺口

#### 缺口 A：有效批次不是按时间最新强约束

当前系统把下一次累计分润的基线建立在：

- `isEffective = true` 的那一批

但没有数据库层强约束保证：

- 同一 `(strategy, billAccount)` 下
- 时间最新的批次一定是有效批次

这意味着：

- 只要存在回填旧日期批次、历史补建批次、手工改状态等情况
- 下一次累计分润就可能基于错误基线

这不是理论风险，当前本地真实库审计中已出现“有效批次时间早于另一批次”的状态。

#### 缺口 B：负向差额时主路径与 helper 语义不一致

当前主路径在负向差额时仍然会直接计算：

- `carryForwardAmount = Δ_t × r_t`
- `paidAmount = distributableAmount`

而另一个 helper 的语义却是：

- 负值不实发
- 缺口结转为待补金额

这说明：

- 代码里实际上存在两套负向场景定义
- 主路径没有统一收敛

#### 缺口 C：个人分配没有形成“累计台账补差”

当前个人层的 `actualTransferAmount` 不是由：

- 个人累计应得
- 减去个人累计已得

推出来的。

当前主路径实际上使用：

- 本批金额按比例拆分
- 再减去“绑定账号净额占本批金额的估算值”

这会带来一个结果：

- 个人层不是严格的累计补差模型
- 而是批次内的即时估算模型

当出现以下情况时，可信度会下降：

- 分润比例调整
- 账号绑定关系调整
- 历史更正跨多个已分润批次
- 需要精确解释“某人累计多拿/少拿多少”

#### 缺口 D：主路径存在明显未接入的设计残留

当前实现里有这些信号：

- `resolveAllocationBaseAmount` 未进入主路径
- `queryPreviousAllocationSumsByParticipant` 未进入主路径
- `cumulativeAllocationBaseAmount` 参数未被真正用于累计个人补差

这通常说明：

- 代码设计曾经考虑过“累计个人对账”
- 但最终没有完整落到主流程

从审计角度看，这不是风格问题，而是业务实现未彻底闭环的信号。

## 7. 实际测试支持到什么程度

### 7.1 已有测试支持的结论

当前后端测试可以支持以下判断：

- `apps/api/test/settlement-allocation.test.ts`
  - 支持“上次留存本次全释放，本次留存只作用于本期差额”
- `apps/api/test/settlement-calculator.test.ts`
  - 支持 helper 层的负值/零值/留存比例边界行为
- `apps/api/test/directional-costs.test.ts`
  - 支持平台抽成退款按方向冲减

### 7.2 当前没有测试支持的结论

以下结论当前没有足够自动化测试支持：

- 主路径在负向差额 + 正留存比例下是否符合业务预期
- 历史更正后，个人层 `actualTransferAmount` 是否等价于累计补差
- 回填旧日期批次后，有效批次是否仍然正确
- 多次历史更正后，累计分润是否仍可审计解释

因此：

- 当前测试证明了“代码现在怎么跑”
- **但还不能证明“代码已经覆盖了所有关键业务边界”**

## 8. 最终判断

### 8.1 当前代码是否“完全错误”

不是。

### 8.2 当前代码是否“已经足够严谨，没有逻辑漏洞”

也不是。

### 8.3 最准确的判断

当前代码：

- 在“全量累计 + 上次留存本次全释放”的特定政策下，主结算公式基本成立
- 但个人层补差、负向差额处理、有效批次基线三个地方，存在足以影响分润结果解释性的逻辑缺口

### 8.4 我的建议

如果这套系统后续还要继续长期跑真实分润：

- **推荐把业务公式提升为显式的累计目标公式**

即：

```text
T_t = C_t - R_t
P_t = T_t - S_{t-1}
```

然后再明确：

- `R_t` 的业务定义
- 个人累计补差的定义
- 有效批次的唯一正确基线规则

这比继续把留存、补差、个人转账三套语义混在当前实现里更科学。
