- `d098307` 旧公式

- 建议的新公式

同时补两组计算：

1. 你刚刚提的简单例子：`100 -> 200`

## 1. 变量和字段映射

| 业务变量 | 含义 | 当前代码字段/来源 |
| --- | --- | --- |
| `C_t` | 当前结算时点累计净额 | `cumulativeNetAmount` |

| `L_t` | 本次结束后新留存余额 | `carryForwardAmount` |

## 2. 三套方案一张总表

| 方案 | 批次头公式 | 个人层公式 | 最终总额是否路径无关 | 个人层是否路径无关 | 对你现在目标的匹配度 | 迁移建议 |
| --- | --- | --- | --- | --- | --- | --- |
| `d098307` 旧公式 | `P_t = C_t × (1-r_t) - S_{t-1}` | `accountHeldAmount = 当前累计目标持有 - 历史已记 held` | 是 | 基本是 | 高 | 可作为回归基线 |
| 当前公式 `5c16113+` | `P_t = L_{t-1} + (C_t - C_{t-1}) × (1-r_t)` | `accountHeldAmount = 当前本批按账号净额占比估算` | 最终清空时是；中间阶段否 | 否 | 中 | 不建议作为长期口径 |
| 建议新公式 | `T_t = C_t × (1-r_t)`；`P_t = max(0, T_t - S_{t-1})` | `个人本次应转 = 个人累计应得 - 个人累计已结` | 是 | 是 | 最高 | 推荐目标方案 |

## 3. 三套方案的核心差异

### 3.1 `d098307` 旧公式

批次头核心是：

```text
当前累计目标已分润额 = C_t × (1-r_t)
本次应分 = 当前累计目标已分润额 - S_{t-1}
```

个人层核心是：

```text
本次个人已持有 = 当前累计目标持有 - 历史累计已持有
本次个人应转 = 本次个人应分 - 本次个人已持有
```

这个版本的特点：

- 批次头天然满足“多次累计 = 一次性”
- 个人层也更接近累计补差

### 3.2 当前公式

当前代码核心是：

```text
Δ_t = C_t - C_{t-1}
本次应分 = L_{t-1} + Δ_t × (1-r_t)
本次新留存 = Δ_t × r_t
```

它的特点：

- 上次留存默认下一次全部释放
- 本次留存只作用于本期差额
- 最终清空留存时，总额能收敛
- 但个人层不是累计补差，所以个人层不收敛

### 3.3 建议新公式

批次头建议明确写成：

```text
T_t = C_t × (1-r_t)
P_t = max(0, T_t - S_{t-1})
```

个人层建议明确写成：

```text
个人累计应得_i = T_t × ratio_i
个人本次应转_i = max/min(个人累计应得_i - 个人累计已结_i)
```

这个版本本质上是：

- 批次头恢复到旧公式方向
- 个人层补足累计已结台账

## 4. 你刚刚那个 `100 -> 200` 例子，直接算

设：

- 第一次累计净额 `C1 = 100`
- 第二次累计净额 `C2 = 200`
- 留存比例 `r = 30%`

### 4.1 第一次分润

无论哪种合理方案，第一次都一样：

- `100 × 0.7 = 70`

所以：

- 第一次已分：`70`
- 第一次留存：`30`

### 4.2 第二次分润，各方案结果

| 方案 | 第二次怎么计算 | 第二次应分 | 两次累计已分 |
| --- | --- | --- | --- |
| 你刚说的 `(200-70)×0.7` | 先减历史已分，再乘比例 | `91` | `161` |
| `d098307` 旧公式 | `200×0.7 - 70` | `70` | `140` |
| 当前公式 | `30 + (200-100)×0.7` | `100` | `170` |

### 4.3 哪个是对的

如果你这里的 `200` 表示：

- **当前累计净额**

那么正确的是：

```text
200 × 0.7 - 70 = 70
```

也就是：

- **先对当前累计净额应用留存比例**
- 再减去历史已分

你刚写的：

```text
(200 - 70) × 0.7 = 91
```

是不对的，因为：

- 它把历史已分额也再次参与了留存比例折算

所以会导致：

- 两次累计已分 = `161`
- 而一次性到第二次时正确累计已分应该是：
  - `200 × 0.7 = 140`

也就是说：

- **它会多分 `21`**

## 7. 迁移建议总表

| 目标 | 建议 |
| --- | --- |
| 不影响现有接口与页面能力 | 只改结算公式与历史重算脚本，不改 API 结构 |
| 满足“多次累计 = 一次性（总额）” | 批次头恢复到 `d098307` 方向 |
| 满足“多次累计 = 一次性（个人）” | 恢复/重建个人累计已持有/已结台账逻辑 |
| 避免当前历史批次语义冲突 | 在新分支实现，并配套历史批次重算 |
| 处理你说的“负未分不分润” | 在新实现里增加 `T_t <= S_{t-1}` 时不创建批次 |

## 8. 最终判断

### 准确回答你的问题

- `d098307` 旧公式：
  - **更合理**
  - 更符合你现在明确提出的业务目标
- 当前公式：
  - **总额最终可收敛**
  - 但**个人层不合理**
- 你刚刚举例的：

```text
(200 - 70) × 0.7
```

  - **不对**
  - 如果 `200` 是当前累计净额，它会多分

### 你真正要的新公式

如果按你现在的目标，真正应该用的是：

```text
当前累计目标已分润额 = 当前累计净额 × (1-r)
本次应分 = max(0, 当前累计目标已分润额 - 上次累计已分润额)
```

不是：

```text
(当前累计净额 - 上次累计已分润额) × (1-r)
```
