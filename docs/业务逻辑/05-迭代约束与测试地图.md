# 迭代约束与测试地图

这篇不是“讲功能怎么用”，而是“指导以后怎么改”。如果未来要继续做程序迭代，先看这一篇能明显降低改坏现有逻辑的概率。

## 1. 改动前必须先认清的硬约束

### 1.1 交易金额设计

- `amount` 在交易表里保存的是绝对值
- `direction` 才代表收入 / 支出 / 不计收支

因此任何改动若只改金额、不改方向，都会破坏统计与分润。

### 1.2 分类是全系统上游

`packages/shared/src/rules.ts` 是最上游的业务口径入口。

一旦改以下内容：

- 主营关键词
- 排除词
- 流量消耗关键词
- 平台抽成关键词
- 退款判定
- 时间窗

会同时影响：

- 导入数量
- 分类结果
- 报表
- 图表
- 分润净额
- 基线测试值

### 1.3 `dedupeKey` 是幂等根基

任何改动以下字段都会改变幂等语义：

- `billAccount`
- `orderId`
- `direction`
- 回退键涉及的时间 / 金额 / 状态 / 描述 / 备注

如果改了 `dedupeKey` 规则，必须评估：

- 老数据是否会与新规则冲突
- 重复导入是否会重写已有交易
- 手动批次编辑是否仍能维持唯一性

### 1.4 增量分润标记是强锁

一旦交易被写入：

- `incrementalSettledAt`
- `incrementalSettlementBatchId`

它会影响：

- 交易能否删除
- 手动批次能否编辑
- 增量分润候选范围

## 2. 当前系统最值得注意的口径分叉

这些不是“潜在风险”，而是当前代码里已经存在的真实差异。

### 2.1 报表纯收益 vs 分润净额

- `apps/api/src/lib/profit-report.ts`
  - `pureProfitSettled` / `pureProfitWithPending`
  - **不扣** `businessRefundExpense`
- `apps/api/src/lib/settlement.ts`
  - 累计 / 增量净额
  - **会扣** `businessRefundExpense`

这意味着：

- “统计页纯收益”与“可分润净额”不是同一个值

### 2.2 图表收益率 vs 报表收益率

- `apps/api/src/routes/transactions.ts`
  - `keyRatios.pureProfitSettled`
  - **会扣**退款支出
  - **固定纳入**关闭交易方向差额
- `apps/api/src/lib/profit-report.ts`
  - **不扣**退款支出
  - 是否纳入关闭交易方向差额受环境变量控制

这意味着：

- 图表页和报表页的“纯收益”当前不是严格同口径

### 2.3 通用 helper 与运行主路径并不一致

- `calculateSettlementAmounts`
  - 当可分润额为负时，会把 `paidAmount` 压到 `0`
- 但预览 / 创建批次实际走的是 `buildSettlementPreview`
  - 当前直接令 `paidAmount = distributableAmount`

也就是说：

- `settlement-calculator.test.ts` 锁住的是 helper 行为
- 但真正运行中的预览 / 创建路径并不完全复用它

后续若想统一结算逻辑，这里是优先排查点。

### 2.4 删除能力在库函数与 HTTP 层不一致

- `apps/api/src/lib/settlement.ts` 里有 `deleteSettlementBatch`
- `apps/api/src/routes/settlements.ts` 的 HTTP DELETE 直接 `403`

所以：

- 内部能力存在
- 外部产品行为是“禁止删除”

以后如果开放删除，不能只改前端按钮。

## 3. 常见改动的影响面矩阵

### 3.1 改导入格式 / 字段兼容

必看文件：

- `apps/api/src/lib/parser.ts`
- `apps/api/src/routes/imports.ts`
- `apps/web/src/App.tsx`（手动批量解析）

建议同步检查：

- `apps/api/test/parser.test.ts`
- `apps/api/test/import-service.test.ts`

### 3.2 改分类关键词 / 业务时间窗

必看文件：

- `packages/shared/src/rules.ts`
- `packages/shared/src/types.ts`

建议同步检查：

- `apps/api/test/filtering.test.ts`
- `apps/api/test/dedupe-internal-transfer.test.ts`
- `apps/api/test/profit-report-baseline.test.ts`

### 3.3 改待发货转成功 / 待到账规则

必看文件：

- `apps/api/src/lib/import-service.ts`
- `apps/api/src/routes/transactions.ts`

原因：

- 导入时 10 天超期转成功
- 图表页也有 10 天自动确认预估

这两个地方目前是耦合常量，不是统一配置源。

### 3.4 改利润公式

必看文件：

- `apps/api/src/lib/profit-report.ts`
- `apps/api/src/lib/settlement.ts`
- `apps/api/src/routes/transactions.ts`
- `apps/api/src/lib/profit-mode.ts`

建议同步检查：

- `apps/api/test/profit-mode.test.ts`
- `apps/api/test/profit-report-baseline.test.ts`
- `apps/api/test/settlement-allocation.test.ts`

### 3.5 改分润分配规则

必看文件：

- `apps/api/src/lib/profit-participants.ts`
- `apps/api/src/lib/settlement.ts`
- `apps/web/src/App.tsx`

建议同步检查：

- `apps/api/test/settlement-calculator.test.ts`
- `apps/api/test/settlement-allocation.test.ts`

### 3.6 改交易删除 / 批次编辑约束

必看文件：

- `apps/api/src/routes/transactions.ts`
- `apps/api/src/routes/imports.ts`
- `apps/api/src/lib/settlement.ts`

这里要特别关注：

- 增量分润标记是否仍然构成锁
- 手动批次是否还能保持一致性
