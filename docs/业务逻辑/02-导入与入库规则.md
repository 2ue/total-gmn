# 导入与入库规则

本文档只描述“交易是如何被系统接纳并变成 `QualifiedTransaction` 的”。它覆盖文件导入、文本导入、手动导入、批次编辑，以及入库前后的关键规则。

## 1. 导入入口

当前后端一共支持四类入口：

- `POST /api/imports`
  - 上传文件，字段名固定为 `file`
  - 同时把原始上传文件保存到 `storage/import-files`
- `POST /api/imports/text`
  - 直接提交文本内容，后端按文本文件处理
- `POST /api/imports/manual`
  - 手动录入，`multipart/form-data`
  - `payload` 为 JSON
  - 允许随每行一起上传截图文件
- `PUT /api/imports/:id/manual-rows`
  - 仅允许编辑 `manual_form` 类型批次
  - 用于补改手动录入批次

## 2. 支持的输入类型

导入源类型定义在 `packages/shared/src/types.ts`：

- `alipay_csv`
- `simple_csv`
- `manual_form`

### 2.1 文件 / 文本解析识别

`apps/api/src/lib/parser.ts` 的识别顺序如下：

1. 先在 `utf8` 与 `gb18030` 之间做最佳努力解码。
2. 如果文本同时包含：
   - `导出信息`
   - `交易时间,交易分类,交易对方`
   
   则按“支付宝原始账单”处理。
3. 否则按“平铺 CSV / 简单表格”处理：
   - 完整支付宝风格字段：至少要有 `交易时间`、`收/支`、`金额`
   - 简化字段：至少要有 `金额`、`备注`
   - 如果 CSV 失败，还会尝试“空白分隔表格”解析

## 3. 字段归一化规则

`toNormalizedTransaction` 会把不同来源的行统一成 `NormalizedTransaction`。

### 3.1 字段映射

- 时间字段优先级：
  - `交易时间`
  - `transactionTime`
  - `时间`
- 金额字段：
  - `金额`
  - `amount`
- 账单账号字段：
  - `账单所属账号`
  - `支付宝账户`
  - `billAccount`
- 说明字段优先级：
  - `商品说明`
  - `description`
  - `备注`
  - `remark`
- 订单号字段：
  - `交易订单号`
  - `orderId`
- 商家订单号字段：
  - `商家订单号`
  - `merchantOrderId`

### 3.2 金额与方向

系统的一个核心设计是：

- `amount` 始终保存为绝对值。
- `direction` 负责表达收入 / 支出 / 不计收支。

方向判定规则：

1. 先读 `收/支` 或 `direction`
   - `收入` → `income`
   - `支出` → `expense`
   - 其他 → `neutral`
2. 如果显式方向无法判定：
   - 且当前状态是 `交易关闭`，并且原始方向列非空，则保持 `neutral`
   - 否则按金额正负推断：
     - 正数 → `income`
     - 负数 → `expense`
     - 零 → `neutral`

### 3.3 默认值与兜底

- 状态默认值：`交易成功`
- 无法解析的时间：回退为当前时间
- 无法解析的账号：回退为当前批次账号，仍没有则记为 `unknown`
- 订单号与商家订单号会去掉所有空白字符

## 4. 业务过滤与分类

分类与过滤规则的唯一事实来源是 `packages/shared/src/rules.ts`。

### 4.1 业务时间窗

只有交易时间 `>= 2025-12-30 00:00:00 +08:00` 的记录才会进入业务口径。

也就是说：

- 早于该时间的记录，直接被过滤
- 这一步发生在分类前

### 4.2 主营识别规则

主营关键词：

- `codex`
- `gemini`
- `90刀`
- `满血api`
- `90d`

排除关键词：

- `批量`
- `批发`

特殊主营说明：

- `闲鱼转账` 被直接视为主营相关描述

主营识别结论：

- 包含主营关键词且不含排除关键词 → 主营
- 描述恰好为 `闲鱼转账` → 主营

### 4.3 分类规则

分类顺序非常关键，当前顺序如下：

1. 描述包含 `闲鱼超级擦亮充值`
   - 分类为 `traffic_cost`
2. 描述包含 `分账-`
   - 分类为 `platform_commission`
3. 状态或描述包含 `退款`
   - 若去掉前缀 `退款-` 后，基础描述能识别为主营
     - 分类为 `business_refund_expense`
   - 否则
     - 分类为 `other_refund`
4. 若状态为 `交易关闭` 且描述可识别为主营
   - 分类为 `closed`
5. 若描述可识别为主营
   - 分类为 `main_business`
6. 否则
   - 分类为 `other`

### 4.4 合格分类

只有以下分类会真正保留入业务表：

- `main_business`
- `manual_add`
- `traffic_cost`
- `platform_commission`
- `business_refund_expense`
- `internal_transfer`
- `closed`

以下分类会被舍弃，但原始行信息可能进入批次 `rawMetaJson`：

- `other_refund`
- `other`

### 4.5 同订单去重与内部互转识别

处理逻辑基于 `orderId`：

- 同一个 `orderId` 只有一条记录时
  - 保留时间最新的一条
- 同一个 `orderId` 同时存在收入与支出时
  - 分别保留最新的收入记录和最新的支出记录
  - 两条都改写为：
    - `category = internal_transfer`
    - `internalTransfer = true`

这一步的意义：

- 内部转账不会污染利润与分润净额
- 但仍然能在明细账本中保留轨迹

## 5. 支付宝导入增强规则

这部分规则只对 `alipay_csv` 生效。

### 5.1 待发货超期自动转成功

在 `apps/api/src/lib/import-service.ts` 中：

- 先对原始解析结果做一次“按现有规则可否进入业务口径”的预筛
- 只有预筛能进入业务口径的记录，才会继续判断是否触发自动转成功

触发条件：

- 来源类型为 `alipay_csv`
- 状态为：
  - `等待发货`
  - `待发货`
- 距交易时间已超过 10 天

触发结果：

- `status` 改为 `交易成功`
- `direction` 改为 `income`

### 5.2 仍处于待发货的支付宝记录直接不导入

自动转成功之后，系统还会再过滤一次：

- 对 `alipay_csv`
- 状态仍为 `等待发货` 或 `待发货`
- 直接从导入流程中剔除

这意味着：

- 超过 10 天的主营候选待发货订单，会被提升为成功收入
- 没超过 10 天，或根本不是业务候选的待发货记录，不会入库

## 6. 去重与幂等

### 6.1 `dedupeKey` 生成规则

如果有 `orderId`：

```text
order | billAccount | orderId | direction
```

如果没有 `orderId`：

```text
fallback | billAccount | transactionTime | direction | amount | status | description | remark
```

所有文本 token 都会先 `trim().toLowerCase()` 后再参与拼接。

### 6.2 单批次内去重

同一次导入过程中，如果多个合格记录算出相同 `dedupeKey`：

- 保留交易时间更晚的一条

### 6.3 与数据库已有记录冲突时

入库前会查库：

- 若数据库没有该 `dedupeKey`
  - 直接创建
- 若数据库已有该 `dedupeKey`
  - 如果库中记录时间 `<=` 新记录时间
    - 用新记录覆盖旧记录的关键字段，并把它挂到当前批次
  - 否则
    - 跳过当前导入记录

因此重复导入的效果不是“叠加”，而是“同键覆盖 / 跳过”。

## 7. 手动导入规则

### 7.1 手动导入固定口径

`importManualTransactions` 会把所有手动行强制写成：

- `sourceType = manual_form`
- `status = 交易成功`
- `category = manual_add`
- `internalTransfer = false`

并且：

- 金额始终取绝对值
- 方向由前端或接口传入
- 描述为空时回退为 `手动添加`
- 账单账号为空时回退为 `unknown`

### 7.2 手动导入订单号

手动导入不会依赖外部订单号，而是生成稳定 `orderId`：

- 格式：`MANUAL{24位摘要}`
- 摘要输入包含：
  - 账号
  - 可选交易时间
  - 方向
  - 金额
  - 描述
  - 截图 sha256
  - 同签名内序号

### 7.3 截图处理

手动导入截图会被保存到：

- `storage/manual-screenshots`

并把以下信息写进交易的 `rawRowJson.screenshot`：

- `fileName`
- `mimeType`
- `size`
- `sha256`
- `storagePath`

同时 `remark` 会写为：

- 有截图：`手动添加截图:<文件名>`
- 无截图：`手动添加`

## 8. 手动批次编辑规则

### 8.1 可编辑前提

只有满足以下条件的批次允许编辑：

- 批次存在
- `sourceType = manual_form`
- 批次内没有任何记录已被增量分润标记

一旦批次内任意一条记录存在：

- `incrementalSettledAt`
- 或 `incrementalSettlementBatchId`

整个批次都会被锁定，不能再批量编辑。

### 8.2 编辑行为

`PUT /api/imports/:id/manual-rows` 的行为不是“补丁式更新”，而是“提交一整份批次最新状态”：

- 传入行里仍保留的 `id`
  - 视为更新
- 原批次中存在但请求中未保留的 `id`
  - 视为删除
- 没有 `id` 的新行
  - 视为新增

### 8.3 编辑期订单号与去重键

- 已有行：沿用原 `orderId`
- 新增行：生成 `MANUALEDIT{24位摘要}`

手动批次编辑的 `dedupeKey` 固定是：

```text
order | billAccount | orderId | direction
```

### 8.4 编辑期校验

- 批次内至少保留一条记录
- 不能引用不属于当前批次的 `id`
- 所有时间必须合法
- 批次内 `dedupeKey` 不能重复
- 新的 `dedupeKey` 不能与其他批次冲突

若全部记录属于同一账号：

- 批次 `billAccount` 更新为该账号

否则：

- 批次 `billAccount` 更新为 `multiple`

## 9. 前端手动批量解析规则

这部分逻辑在 `apps/web/src/App.tsx`，发生在真正调用 `/api/imports/manual` 之前。

### 9.1 支持的分隔方式

- 制表符
- 英文逗号
- 中文逗号
- 多空格 / 空白分隔

### 9.2 支持的表头别名

时间列：

- `时间`
- `交易时间`
- `datetime`
- `transactiontime`
- `date`

账号列：

- `账号`
- `账单账号`
- `账单所属账号`
- `支付宝账户`
- `billaccount`
- `account`

金额列：

- `金额`
- `amount`
- `money`

说明列：

- `说明`
- `备注`
- `description`
- `remark`
- `商品说明`
- `content`

### 9.3 无表头时的位置规则

- 2 列：`金额 + 说明`
- 3 列：
  - 若第 1 列像时间：`时间 + 金额 + 说明`
  - 否则：`金额 + 说明 + (时间或账号)`
- 4 列：`金额 + 说明 + 时间 + 账号`
- 5 列及以上：
  - 优先尝试从尾部识别 `日期 + 时间 + 账号`
  - 识别失败时退回到：
    - 第 1 列金额
    - 最后 2 列时间与账号
    - 中间拼接为说明

### 9.4 金额与方向

- 金额允许带 `,`、空格、`¥`、`￥`
- 批量解析阶段会根据金额正负推断方向
  - 负数 → `expense`
  - 正数 → `income`
  - 0 或非法 → 回退默认方向
- 真正提交给后端时金额会被转成正数文本
