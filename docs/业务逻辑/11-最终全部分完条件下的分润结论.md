# 最终全部分完条件下的分润结论

本文档专门回答这一个条件下的问题：

> 如果最后一次分润会把剩余留存全部分完，那么多次分润的累计结果，是否会和一次性分润一致？

这个问题和前面“每一轮都按固定留存比例看阶段结果”不是同一个问题，所以单独说明。

---

## 1. 结论先说

### 1.1 只看批次总额

如果满足下面两个条件：

1. 分润批次链本身是自洽的  
   （也就是上一批记录满足 `累计净额 = 累计已分润 + 留存余额`）
2. 最后一次会把剩余留存全部分完  
   （等价于最后一次 `carryRatio = 0`，且没有新的净额变化）

那么：

- **当前代码在批次总额层面，最终不会多分，也不会少分**
- **你前面提出的“当前未分余额统一留存”方案，在批次总额层面最终也不会多分，也不会少分**

最终累计已分润都会收敛到：

- **当前最终累计净额**

### 1.2 但如果看个人层

即使在“最后一次全部分完”的条件下：

- **当前代码的个人层累计结果，仍然不一定等于一次性分润**

也就是说：

- 批次头总额可以最终对上
- 但每个分润者实际该转多少、该扣多少，累计起来仍可能和一次性结果不一致

### 1.3 另外，当前真实库还有一个独立风险

当前真实开发库里：

- 有效批次 `isEffective = true` 的那一批
- 不是时间更晚的那一批

如果历史上两批都已经实际付过钱，而系统却只拿较早那批做基线，那么：

- **下一次分润仍然可能真实多分**

所以要区分两个层次：

1. **公式在理想链条里是否最终收敛**
2. **你当前真实库状态是否会因为基线选错而实际多付**

---

## 2. 批次头总额：真实数据实测

### 2.1 测试输入

使用真实项目数据：

- 第一阶段累计净额：`35923.79`
- 导入后最终累计净额：`125859.77`
- 中间留存比例：`30%`
- 最后一轮：全部分完（`carryRatio = 0`）

### 2.2 当前代码方案

按当前代码主路径：

- 第一轮分润：`25146.65`
- 第一轮留存：`10777.14`
- 第二轮分润：`73732.33`
- 第二轮留存：`26980.79`
- 第三轮最终清空留存：`26980.79`

三轮累计已分润总额：

- `25146.65 + 73732.33 + 26980.79 = 125859.77`

与最终累计净额：

- `125859.77`

完全一致。

### 2.3 “当前未分余额统一留存”方案

按你前面那版方案：

- 第一轮分润：`25146.65`
- 第一轮留存：`10777.14`
- 第二轮分润：`70499.18`
- 第二轮留存：`30213.94`
- 第三轮最终清空留存：`30213.94`

三轮累计已分润总额：

- `25146.65 + 70499.18 + 30213.94 = 125859.77`

也与最终累计净额：

- `125859.77`

完全一致。

### 2.4 这说明什么

说明在“最后一次全部分完”的前提下：

- **前面提到的 `10777.14` 和 `7543.99`，不是最终总额多分**
- 它们只是：
  - 中间阶段的留存释放路径不同
  - 导致中间各轮分润额不同

最终清空后：

- 批次头总额是可以收敛一致的

---

## 3. 那前面为什么会看起来“多分了”？

因为前面比较的是：

- 在某一个中间阶段，
- 留存比例仍然保留，
- 还没有最终清空，
- 直接把“分两次后的累计已分润”与“一次性按同样留存比例分”的结果做比较。

这种比较下：

- 当前代码会显得多分
- 旧新方案也会显得多分

但那不是最终总额口径，而是：

- **阶段性累计已分润口径**

一旦把最后的留存全部分完：

- 批次头总额就会对齐到最终累计净额

---

## 4. 当前代码在“最终全部分完”条件下还有没有漏洞

### 4.1 批次头总额层

在理想前提下：

- **没有最终总额重复计算**
- **也没有最终总额漏计算**

换句话说：

- 当前代码的批次头总额层，最终是收敛的

### 4.2 个人层

但个人层结论不一样。

即使最终全部分完，当前代码的个人层累计结果仍然**不一定等于一次性分润**。

原因是：

- 每一轮的 `actualTransferAmount`
- 不是按“个人累计应得 - 个人累计已结”算的
- 而是按“本批金额 + 当前账号持有估算”算的

这会导致：

- 多轮计算时，个人层路径不具备严格无关性

## 5. 个人层：真实数据实测

### 5.1 当前代码多轮 + 最终全部分完

按真实数据、真实分润者配置，当前代码三轮累计到个人层的结果是：

- `2429886596@qq.com`
  - `amount = 34611.44`
  - `actualTransferAmount = 28359.16`
- `k佬`
  - `amount = 22025.46`
  - `actualTransferAmount = 22025.46`
- `chunqingdepaigu@163.com`
  - `amount = 69222.87`
  - `actualTransferAmount = -50384.62`

### 5.2 一次性最终全部分完

如果直接在最终账本上一次性全部分完，当前代码个人层结果是：

- `2429886596@qq.com`
  - `amount = 34611.44`
  - `actualTransferAmount = 30443.81`
- `k佬`
  - `amount = 22025.46`
  - `actualTransferAmount = 22025.46`
- `chunqingdepaigu@163.com`
  - `amount = 69222.87`
  - `actualTransferAmount = -52469.27`

### 5.3 差异

对比后可以看到：

- `2429886596@qq.com`
  - 多轮累计 `actualTransferAmount` 比一次性少 `2084.65`
- `chunqingdepaigu@163.com`
  - 多轮累计 `actualTransferAmount` 比一次性多 `2084.65`
- `k佬`
  - 一致

这说明：

- **当前代码在个人层不满足“多次累计结果 = 一次性结果”**

### 5.4 原因

根因不是批次头总额，而是当前个人层公式：

- 当前每轮都重新按账号净额占比估算 `accountHeldAmount`
- 然后再算 `actualTransferAmount`

所以：

- 每轮的“账号已持有”会被重复按本轮基数重映射
- 累计起来不等于一次性最终映射

## 6. “个人累计应得 / 已得台账”是什么意思

这不是单纯为了看报表，也不只是“增加一些统计”。

它的真正作用是：

- 让**每个分润者**也满足路径无关性

如果要做到这一点，系统至少要在逻辑上能表达：

- 某个分润者截至当前时点的**累计应得**
- 某个分润者截至当前时点的**累计已结**
- 本次个人补差：

```text
个人本次应转 = 个人累计应得 - 个人累计已结
```

当前代码没有这样一套显式累计个人台账。

所以才会出现：

- 批次头总额最终能对上
- 个人层最终却对不上

---

## 7. 当前真实库的独立风险

### 7.1 现状

当前真实开发库中：

- `2026-01-01` 那批是有效批次
- `2026-02-01` 那批不是有效批次

但从时间上看：

- `2026-02-01` 才是更晚一批

### 7.2 这意味着什么

如果历史上两批都是真实执行过的付款，

而系统下一次只拿：

- `2026-01-01` 那批

作为基线，那么：

- **现实世界里会发生多付**

前面审计里已经测过：

- 仅这个基线错误，就会带来 `9166.58` 的偏差

所以：

- “公式最终是否收敛”
- 和“当前真实库现在会不会实际多付”

是两个不同问题。

当前真实库这块风险依然存在。

---

## 8. 最终结论

### 结论 1：批次头总额

如果最后一次会把留存全部分完，而且批次链本身自洽：

- 当前代码不会在**最终批次头总额**上重复计算
- 也不会漏计算

### 结论 2：个人层

当前代码在**个人层**不满足：

- “多次累计结果 = 一次性结果”

这是当前实现的真实漏洞。

### 结论 3：当前真实库

当前真实库还存在另一个独立风险：

- 有效批次基线异常

这会让下一次真实分润，即使最终会清空，也可能基于错误历史已分额起算。

### 结论 4：你的目标

如果你的硬要求是：

- 批次头总额路径无关
- 个人层也路径无关

那么当前代码还不够。

需要至少补：

1. 正确的有效批次基线约束
2. 个人累计应得 / 已结的补差逻辑
