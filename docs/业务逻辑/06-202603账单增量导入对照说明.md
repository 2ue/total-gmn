## 2. 先给结论

### 2.1 `等待对方付款`，旧逻辑会不会入库？

结论：**会，但有前提**。

旧逻辑里：

- 只会过滤支付宝账单中的：
  - `等待发货`
  - `待发货`
- **不会过滤**：
  - `等待对方付款`
  - `等待对方确认收货`
  - `等待确认收货`

所以：

- 如果一条记录本身已经能被识别为业务记录（例如描述里带 `Codex` / `Gemini` 等关键词）
  - 那么 `等待对方付款` **会入库**
- 但如果一条记录描述只是 `订单9123645` 这种，旧逻辑识别不出是业务
  - 即使状态是 `等待对方付款`，**也不会入库**

这正是这次 `SMM*` 规则要补的缺口。

### 2.2 这次导入后覆盖了老数据，是否正常？

结论：**是正常覆盖，而且覆盖行为符合当前系统设计**。

本次导入前：

- `QualifiedTransaction` 总数：`8695`

本次导入后：

- `QualifiedTransaction` 总数：`23233`
- 净增：`14538`

看起来新批次入了 `22899` 条，但总量只增加了 `14538`，原因不是丢数据，而是：

- 大量旧记录被新账单中的同一交易覆盖更新了
- 覆盖后的记录会被挂到新批次名下

这就是当前系统的幂等设计。

### 2.3 旧的 `20251116-20260216-wx` 批次为什么只剩 11 条？

结论：**因为新账单覆盖了 2026-01-05 到 2026-03-05 的同键记录，剩下的 11 条正好都在新账单覆盖范围之前。**

这说明：

- 新账单覆盖区间内的旧数据都被正常更新到新批次
- 覆盖区间外的数据保留下来

### 2.4 会引起增量分润问题吗？

就当前本地开发库来说：**不会产生现成的增量分润问题**。

原因是当前库里：

- 没有任何增量分润批次
- `incrementalSettledAt` / `incrementalSettlementBatchId` 标记数量为 `0`

也就是说：

- 目前不存在“已增量结算的数据被重复算入”的历史风险

但要注意：

- 当前库里存在累计分润批次
- 导入了大量历史重叠数据之后，旧累计批次的业务含义已经过时

所以：

- **增量分润没有历史冲突**
- **累计分润的历史结果如果要继续作为正式参考，建议重新评估 / 重建**

### 2.5 如果一条交易之前是成功，后来被退款，会不会覆盖旧成功？

结论：**通常不会覆盖，而是作为“新的退款事件”单独入账。**

当前系统是“流水账模型”，不是“订单最终状态模型”。

也就是说：

- 原交易成功 → 仍然保留
- 后续退款 → 作为新的退款记录再入一条
- 后续手续费退回 → 作为新的退费记录再入一条

这套模型更适合做对账和利润分析。

对于你说的“我们的服务没有退货”：

- 系统不会主动凭空制造退货
- 但只要账单里出现了退款 / 关闭 / 退费，系统会按账单事实入账
- 这些通常代表平台侧、支付侧或异常取消，不一定是你主动做了“退货”

## 3. 旧解析逻辑（本次改动前）

这一节的“旧逻辑”，指的是本次补规则之前的代码行为。

### 3.1 旧逻辑识别业务单的依据

旧逻辑只靠**描述文本关键词**识别主营：

- `codex`
- `gemini`
- `90刀`
- `满血api`
- `90d`

排除词：

- `批量`
- `批发`

特殊主营：

- 描述刚好等于 `闲鱼转账`

这意味着旧逻辑的核心特点是：

- 看“商品说明 / 备注”像不像业务
- **不看**商家订单号模式
- **不看**对方账号特征

### 3.2 旧逻辑下哪些会被当成主营

例如这些会被识别：

- `CODEX 满血API 月卡每日90刀`
- `Codex月卡每日90刀 ai编程`
- `Gemini3.0 API月卡资源分享`

### 3.3 旧逻辑下哪些会被漏掉

以下这类旧逻辑识别不了：

- 描述只是 `订单9123645`
- 但实际商家订单号是 `SMMD...`
- 或 `SMMB...`
- 或 `SMMC...`

原因很简单：

- 描述里没有 `Codex / Gemini / 月卡关键词`
- 旧逻辑又不看 `merchantOrderId`

### 3.4 旧逻辑对 `sys***@alipay.com` 的处理

旧逻辑没有把 `sys***@alipay.com` 当成特殊对象。

因此：

- 描述是 `服务费[...]`
- 对方账号是 `sys***@alipay.com`

这类记录旧逻辑不会落到：

- `platform_commission`

而是会落到：

- `other`

最终：

- **不会入库**

同理：

- `退费成功 + sys***@alipay.com + 服务费退回(...)`

旧逻辑也不会被正确入库。

### 3.5 旧逻辑对状态的处理

旧逻辑里：

- `等待发货` / `待发货`
  - 支付宝账单中会被特殊处理
  - 超过 10 天且已通过业务候选预筛的，会转成 `交易成功 + income`
  - 其余仍是待发货的，会被过滤掉，不入库
- `等待对方付款`
- `等待对方确认收货`
- `等待确认收货`
  - 如果已经被识别为主营
  - 会按待到账主营收入入库

### 3.6 旧逻辑的覆盖规则

旧逻辑的覆盖规则与现在一样，核心是 `dedupeKey`。

有 `orderId` 时：

```text
order | billAccount | orderId | direction
```

所以旧逻辑下只要满足：

- 账单账号相同
- `orderId` 相同
- `direction` 相同

新导入记录就会命中同一键。

之后如果：

- 库里旧记录时间 `<=` 新记录时间

则会把以下字段整体更新成新的：

- `transactionTime`
- `orderId`
- `merchantOrderId`
- `description`
- `direction`
- `amount`
- `status`
- `category`
- `internalTransfer`
- `billAccount`
- `remark`
- `rawRowJson`
- `batchId`

### 3.7 旧逻辑的退款问题

旧逻辑对退款也主要靠“描述关键词”。

能识别成主营退款的前提：

- 状态或描述包含 `退款`
- 并且去掉 `退款-` 后的描述，仍然能靠关键词识别出主营

这意味着：

- `退款-CODEX 满血API 月卡每日90刀`
  - 能识别为 `business_refund_expense`
- `退款-订单9101955`
  - 旧逻辑识别不了主营
  - 不会进入 `business_refund_expense`

### 3.8 旧逻辑的统计偏差

旧逻辑即使把某些手续费 / 成本类退款入库了，也有一个统计问题：

- `platform_commission`
- `traffic_cost`

在报表 / 图表 / 分润里是按绝对值直接累加的。

这会导致：

- 手续费支出 `10`
- 手续费退回 `2`

旧逻辑统计结果会变成：

- 手续费成本 `12`

而不是正确的：

- 手续费净成本 `8`

## 4. 现在的解析逻辑（本次改动后）

### 4.1 新增规则一：`sys***@alipay.com` 按收支入库

现在只要满足：

- `rawRowJson["对方账号"] === "sys***@alipay.com"`

就直接归类为：

- `platform_commission`

并保留账单原始 `direction`：

- `expense` → 手续费支出
- `income` → 手续费退回

这次新账单中实际识别到：

- `462` 条 `交易成功 + expense`
- `2` 条 `退费成功 + income`

### 4.2 新增规则二：整组 `SMM*` 商家单号按业务单解析

虽然你最初提出的是 `SMMD`，但实际账单分析发现：

- 退款和关闭并不只出现在 `SMMD`
- 还出现在 `SMMB`、`SMM1` 等前缀

因此当前实现不是只认 `SMMD`，而是认：

- `merchantOrderId.startsWith("SMM")`

原因是数据事实表明这是一整组同业务家族单号。

### 4.3 `SMM*` 现在的分类逻辑

对于 `merchantOrderId` 以 `SMM` 开头的记录：

- 若状态 / 描述包含 `退款`
  - 归类为 `business_refund_expense`
- 若状态为 `交易关闭`
  - 归类为 `closed`
- 其他
  - 归类为 `main_business`

这意味着现在像下面这些都能进入正确口径：

- `订单9123645` + `SMMD...` → `main_business`
- `退款-订单9101955` + `SMMB...` → `business_refund_expense`
- `订单9017145` + `SMM1...` + `交易关闭` → `closed`

### 4.4 现在 `等待对方付款` 的真实行为

现在有两类情况：

#### A. 关键词型业务单

例如：

- `CODEX 满血API 月卡每日90刀`

旧逻辑就能入库，现在也一样能入库。

#### B. `SMM*` 订单型业务单

例如：

- 描述只是 `订单9123645`
- 商家订单号是 `SMMD...`
- 状态是 `等待对方付款`

旧逻辑：

- 不会入库

现在：

- 会作为 `main_business + income + 等待对方付款` 入库

这次新批次里实际有：

- `5` 条 `SMM* + 等待对方付款 + main_business`

### 4.5 现在手续费 / 流量成本的退款会冲减成本

本次同时修正了统计逻辑：

- `platform_commission`
- `traffic_cost`

现在会按 `direction` 做净额处理：

- `expense` → 增加成本
- `income` → 冲减成本

因此：

- 手续费 `expense 73.99`
- 手续费退回 `income 0.27`

最终报表中的 `platformCommission` 会是：

- `73.72` 的净效果

而不是旧逻辑那种：

- `74.26` 全部当成本累加

## 6. 覆盖逻辑：什么会覆盖，什么不会覆盖

### 6.1 会正常覆盖的情况

只要满足同一个：

- `billAccount`
- `orderId`
- `direction`

就会命中同一 `dedupeKey`。

因此这类状态变化会覆盖：

- `等待对方付款` → `交易成功`
- `等待对方确认收货` → `交易成功`
- `等待确认收货` → `交易成功`

前提是：

- 新账单里的记录时间不早于库里旧记录

在这类情况下，覆盖是正确且符合预期的。

### 6.2 不会覆盖、而会新增一条的情况

以下情况通常不会覆盖，而是新增事件：

- 退款行有新的 `orderId`
- 手续费退回行有新的 `orderId`
- 方向发生变化，导致 `dedupeKey` 不同

这也是当前系统有意采用的“流水账模型”。

### 6.3 为什么这对退款更合理

因为退款本质上不是“把旧交易改成失败”，而是：

- 先有一笔成交
- 后有一笔退款

这两件事都应该保留在账上。

当前系统就是按这个思路处理。
