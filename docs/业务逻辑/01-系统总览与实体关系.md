# 系统总览与实体关系

本文档用于建立对项目整体的“稳定脑图”。如果只想先搞清楚系统怎么流转、核心对象有哪些、每个模块负责什么，先读这一篇。

## 1. 系统目标

项目目标不是做一个通用财务系统，而是围绕“闲鱼 / 支付宝账单 → 业务交易 → 统计报表 → 分润批次”这条链路，提供一套可持续迭代的运营后台。

当前系统已经覆盖的核心能力：

- 导入支付宝账单、简化 CSV、文本内容、手动录入记录。
- 把原始账单归一成统一交易结构。
- 按业务关键词与规则筛出真正参与经营口径的交易。
- 输出交易明细、聚合图表、利润报表。
- 基于分润者比例与留存比例生成累计 / 增量分润批次。

## 2. 模块分工

### 2.1 Monorepo 结构

- `apps/web`
  - 单页前端。
  - 负责导入页面、明细页、图表页、统计页、分润设置页、累计分润页。
- `apps/api`
  - 唯一后端服务。
  - 负责解析导入、业务分类、报表统计、分润计算、批次落库。
- `packages/shared`
  - 放前后端共享的业务基础定义。
  - 包含 `SourceType`、`Category`、`NormalizedTransaction`、分类规则与时间解析。

### 2.2 后端核心模块

- `apps/api/src/lib/parser.ts`
  - 负责“读得懂输入”。
  - 包括编码识别、账单类型识别、字段抽取、方向与金额初步归一化。
- `apps/api/src/lib/import-service.ts`
  - 负责“让导入能落地”。
  - 包括待发货超期转成功、过滤不可导入记录、构建 `dedupeKey`、导入批次落库。
- `packages/shared/src/rules.ts`
  - 负责“什么算业务交易”。
  - 包括主营识别、退款识别、内部互转识别、业务时间窗过滤。
- `apps/api/src/lib/profit-report.ts`
  - 负责利润报表汇总与钻取明细。
- `apps/api/src/routes/transactions.ts`
  - 负责明细页、汇总卡片、图表页所需接口。
- `apps/api/src/lib/profit-participants.ts`
  - 负责分润者配置校验与存储。
- `apps/api/src/lib/settlement.ts`
  - 负责累计 / 增量分润预览、分配、批次创建与有效批次切换。

## 3. 页面与后端能力映射

前端在 `apps/web/src/App.tsx` 中集中管理页面状态，当前主要页签如下：

- 导入页
  - 对应 `/api/imports`、`/api/imports/text`、`/api/imports/manual`
  - 也包含手动批次回显与批量编辑能力。
- 明细页
  - 对应 `/api/transactions`、`/api/transactions/:id`、`/api/transactions/:id/screenshot`
- 图表页
  - 对应 `/api/transactions/summary`、`/api/transactions/charts`
- 统计页
  - 对应 `/api/reports/profit`、`/api/reports/profit/details`
- 分润设置页
  - 对应 `/api/profit-participants`、`/api/accounts`
- 累计分润页
  - 对应 `/api/settlements/preview`、`/api/settlements`
- 增量分润
  - 后端已支持，前端页签代码保留但默认隐藏。

## 4. 核心数据实体

### 4.1 `ImportBatch`

含义：一次导入动作的批次记录。

关键字段：

- `sourceType`
- `fileName`
- `billAccount`
- `rawMetaJson`
- `importedAt`

职责：

- 记录“这批数据从哪里来”。
- 保存解析阶段的辅助信息，例如：
  - 编码
  - parser 名称
  - 总解析行数
  - 合格/不合格行数量
  - 未入库原始行
  - 上传文件元信息

### 4.2 `QualifiedTransaction`

含义：真正进入业务口径的交易记录。

关键字段：

- `transactionTime`
- `billAccount`
- `direction`
- `amount`
- `status`
- `category`
- `internalTransfer`
- `dedupeKey`
- `incrementalSettledAt`
- `incrementalSettlementBatchId`

关键事实：

- 它是整个系统的业务基表。
- 报表、图表、分润全部围绕它展开。
- 金额是正数，收支由 `direction` 表达。
- 同一业务记录是否重复，最终由 `dedupeKey` 决定。

### 4.3 `ProfitParticipant`

含义：分润参与方配置。

关键字段：

- `name`
- `billAccount`
- `ratio`
- `note`

特点：

- `billAccount` 可为空，表示不绑定具体账号。
- 一个账单账号最多绑定一个分润者。
- 比例以 6 位小数存储。

### 4.4 `SettlementBatch`

含义：一次分润结算的批次头。

关键字段：

- `strategy`
- `billAccount`
- `settlementTime`
- `carryRatio`
- `periodNetAmount`
- `previousCarryForwardAmount`
- `cumulativeNetAmount`
- `settledBaseAmount`
- `distributableAmount`
- `paidAmount`
- `carryForwardAmount`
- `cumulativeSettledAmount`
- `isEffective`

关键事实：

- `billAccount=""` 代表全局口径，不是缺失值。
- 同一 `(strategy, billAccount)` 维度只允许一个有效批次。

### 4.5 `SettlementAllocation`

含义：某次分润批次下，每个分润者的分配结果。

关键字段：

- `participantId`
- `participantName`
- `participantBillAccount`
- `ratio`
- `amount`
- `accountHeldAmount`
- `actualTransferAmount`

职责：

- 保存“按比例应分多少”。
- 保存“该分润者绑定账号本轮已持有多少”。
- 保存“本次实际应转多少 / 应回补多少”。

## 5. 端到端流程

```text
原始输入
  ├─ 支付宝原始账单
  ├─ 简化 CSV / 文本
  └─ 手动录入（可带截图）
        │
        ▼
字段归一化（parser.ts）
        │
        ▼
业务过滤与分类（rules.ts）
        │
        ▼
导入增强与去重（import-service.ts）
        │
        ▼
QualifiedTransaction
  ├─ 明细 / 图表
  ├─ 利润报表
  └─ 分润预览 / 分润批次
```

## 6. 系统级不变量

这些不变量是后续改动时最容易被破坏、也最需要保护的部分：

- 金额绝对值与收支方向分离。
- 真正参与业务口径的是“分类后的合格交易”，不是所有原始导入行。
- `internal_transfer` 记录要留在账本里，但不应混入利润与分润净额。
- 增量分润一旦给交易打上 `incrementalSettledAt` / `incrementalSettlementBatchId`，相关交易与手动批次都会进入受限状态。
- 前端多个页面虽然都在展示“统计”，但底层并不是完全一套公式，不能想当然地一起改。
